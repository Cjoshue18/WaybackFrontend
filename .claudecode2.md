1. Módulo de Perfil e Identidad (/api/profile)Endpoint de Lectura/Escritura: GET y PUT hacia {{BackEnd_Https}}/api/profile/mi-perfil. Estructura del JSON (Payload):cliNombre (String): Nombres del cliente.  cliApellido (String): Apellidos del cliente.  usuUsername (String): Nombre de usuario.  usuEmail (String): Correo electrónico vinculado.  cliTelefono (String / Null): Teléfono de contacto.  2. CRUD de Direcciones (/api/profile/direcciones)Endpoints:GET /api/profile/direcciones: Lista todas las direcciones del usuario autenticado.  POST /api/profile/direcciones: Registra una nueva dirección.  PUT /api/profile/direcciones/:id: Edita una dirección existente.  DELETE /api/profile/direcciones/:id: Elimina un registro de dirección.  Campos del Payload: DirCalle, DirDistrito, DirProvincia, DirDepartamento, DirReferencia, y un booleano DirPreferido (para marcar cuál es la dirección de envío por defecto).  Regla de Negocio Crítica: El borrado y edición están protegidos en el backend mediante el token del cliente. Si un cliente intenta alterar un :id de dirección que no le pertenece, el sistema rechazará la solicitud de forma estricta.  3. Catálogo, Variantes y Estructura de Filtros CombinadosRefactor de URLs: Se debe eliminar por completo el formato estático antiguo (ej. /productos/categorias=5). A partir de ahora, todo debe realizarse a través de Query Strings (?) sobre la ruta base de /productos para que los filtros puedan combinarse dinámicamente entre sí.  Parámetros de Búsqueda Soportados (Query Params):categoria: Permite acumular múltiples valores en la barra de direcciones (?categoria=5&categoria=6).  estilo: Soporta selección múltiple idéntica a las categorías (?estilo=X&estilo=Y).  color y talla: Permiten concatenar múltiples selecciones en el catálogo (?color=rojo&color=verde, ?talla=s&talla=m).  precioMin y precioMax: Control de rangos numéricos.  stock: Booleano (?stock=true) para filtrar y renderizar únicamente ítems con stock disponible en almacén.  Restricción del Filtro de Género: A diferencia de los demás filtros, el género (Hombre, Mujer, Unisex) debe configurarse como un control de selección única en el frontend. No se permite combinar múltiples opciones simultáneamente para evitar redundancia en el catálogo.  Endpoint de Variantes para Modales: GET a /api/admin/reportes/productos/:id/variantes (o integrado en la consulta individual de productos). Retorna un arreglo con campos esenciales para la ventana emergente de selección previa a la compra: varId, varTalla, colorNombre, colorHex (código hexadecimal del color para las muestras visuales) y varStock.  4. Lógica de Descuentos por Fechas (Frontend)El JSON que te entrega el endpoint de productos incluye los siguientes campos para el control de ofertas temporales:proPrecio (Número decimal): Precio base del producto.  proDescuento (Número entero, ej. 20): Porcentaje de rebaja a aplicar.  proDescuentoInicio y proDescuentoFin (Strings con formato ISO de fecha/hora, ej. 2026-06-09T00:00:00).  Algoritmo a Programar: Se debe verificar mediante código si la fecha actual del sistema se encuentra dentro del rango de inicio y fin. Si la condición es verdadera, se aplica el descuento matemático al componente visual (precio - (precio * (descuento / 100))). En la interfaz se renderizará el precio original tachado (ej. ~~S/ 40.00~~) junto al precio final rebajado en texto resaltado (ej. S/ 30.00).  5.
